<project_specification>
  <project_name>LlamaExtract CI/CD Demo with Webhook Integration</project_name>

  <overview>
    A minimal Python project that demonstrates three things:
    1. Using LlamaExtract to extract structured data (merchant name, date, total) from receipt PDFs.
    2. A CI/CD pipeline (GitHub Actions) that re-runs extraction whenever the Pydantic schema changes in a PR, computes field-level accuracy against ground truth, and posts a PR comment showing the results and delta vs. the previous baseline.
    3. Webhook integration built into extract.py — when WEBHOOK_URL is set, extraction jobs include
       webhook configuration so LlamaExtract POSTs completion events to the configured endpoint
       (e.g., webhook.site) for real-time visualization.

    The project is intentionally simple — a handful of Python scripts, a GitHub Action workflow, and sample data. No web framework, no database, no frontend.
  </overview>

  <data_model>
    <schema name="InvoiceSchema" file="schema.py">
      This is the Pydantic model that defines what fields to extract from receipts.
      When a developer modifies this file and opens a PR, the CI pipeline re-runs extraction and evaluation.

      ```python
      from pydantic import BaseModel, Field
      from typing import Optional

      class InvoiceSchema(BaseModel):
          """Schema for extracting key fields from receipt/invoice images."""
          merchant_name: Optional[str] = Field(
              default=None,
              description="The name of the store or merchant on the receipt"
          )
          date: Optional[str] = Field(
              default=None,
              description="The date of the transaction, in YYYY-MM-DD format"
          )
          total: Optional[str] = Field(
              default=None,
              description="The total amount charged, including currency symbol if present"
          )
      ```

      All fields are Optional[str] so that missing extractions return None rather than causing validation errors.
      The description fields guide LlamaExtract on what to look for and how to format it.
    </schema>

  <scripts>
    <script name="extract.py" description="Run LlamaExtract on all PDFs and save results">
      Purpose: Upload each PDF in data/ to LlamaExtract, run extraction with the schema from schema.py,
      poll for completion, and save results to results/extraction_output.json.

      Flow:
      1. Load LLAMA_CLOUD_API_KEY from environment (with dotenv fallback for local dev).
      2. Load WEBHOOK_URL from environment (optional).
      3. Import InvoiceSchema from schema.py.
      4. For each PDF in data/:
         a. Upload file: client.files.create(file=path, purpose="extract")
         b. Build extraction kwargs:
            extract_kwargs = dict(
                file_input=file_obj.id,
                configuration={
                    "data_schema": InvoiceSchema.model_json_schema(),
                    "extraction_target": "per_doc",
                    "tier": "agentic",
                }
            )
         c. If WEBHOOK_URL is set, add webhook configuration:
            extract_kwargs["webhook_configurations"] = [
                {
                    "webhook_url": webhook_url,
                    "webhook_events": ["extract.success", "extract.error"],
                    "webhook_output_format": "json"
                }
            ]
            Print: "Webhook configured: events will be sent to {webhook_url}"
         d. Create extraction job: client.extract.create(**extract_kwargs)
         e. Poll for completion (sleep 2s between checks):
            while job.status not in ("COMPLETED", "FAILED", "CANCELLED"):
                time.sleep(2)
                job = client.extract.get(job.id)
         f. On COMPLETED: extract the result JSON from job.extract_result.
         g. On FAILED/CANCELLED: log error, record null result for that file.
      5. Save all results to results/extraction_output.json:
         {
           "invoice_1.pdf": {"merchant_name": "...", "date": "...", "total": "..."},
           "invoice_2.pdf": {"merchant_name": "...", "date": "...", "total": "..."},
           ...
         }
      6. Print summary to stdout.
         If WEBHOOK_URL was set, remind user: "Check {webhook_url} in your browser to see the webhook payloads."

      CLI usage:
        python extract.py                        # Extract all PDFs
        python extract.py --init-ground-truth     # Extract and save as ground_truth.json (first run only)

      When --init-ground-truth is passed, the output is written to ground_truth.json instead of
      results/extraction_output.json. The user should review and correct this file before committing.
    </script>

    <script name="evaluate.py" description="Compare extraction results against ground truth">
      Purpose: Load extraction results and ground truth, compare field-by-field, compute accuracy,
      optionally compare against baseline metrics, and output a summary.

      Accuracy computation:
      - For each file in ground truth, compare each field (merchant_name, date, total).
      - A field is "correct" if the extracted value matches the ground truth value (case-insensitive string comparison after stripping whitespace).
      - A field is "missing" if the extracted value is None/null but ground truth has a value. This counts as INCORRECT.
      - A field is "wrong" if the extracted value is non-null but does not match ground truth. This counts as INCORRECT.
      - A field is "extra" if extracted has a value but ground truth is None. This is ignored (not counted).
      - overall_accuracy = correct_fields / total_ground_truth_fields
      - Example: if GT has 3 fields with values for a file, extraction returns only 1 correct field:
        accuracy for that file = 1/3 = 33.3%

      Flow:
      1. Load ground_truth.json and results/extraction_output.json.
      2. For each file, compare each field.
      3. Compute per-file accuracy and overall accuracy.
      4. Save new metrics to results/current_metrics.json.
      5. If metrics.json exists (baseline), compute delta:
         - delta_accuracy = new_accuracy - baseline_accuracy
         - per-field changes (was wrong, now correct; was correct, now wrong; etc.)
      6. Print results table to stdout using tabulate.
      7. If --output-markdown flag is passed, also write results as a markdown string to
         results/pr_comment.md for the GitHub Action to post.

      Markdown output format (results/pr_comment.md):
      ```
      ## Extraction Evaluation Results

      **Overall Accuracy: 88.9%** (delta: +5.6% vs baseline)

      | File | Accuracy | merchant_name | date | total |
      |------|----------|---------------|------|-------|
      | invoice_1.pdf | 100% | correct | correct | correct |
      | invoice_2.pdf | 66.7% | correct | correct | wrong |
      | invoice_3.pdf | 100% | correct | correct | correct |

      ### Changes from Baseline
      - invoice_2.pdf: total changed from wrong to correct (+11.1%)
      - invoice_2.pdf: date changed from correct to wrong (-11.1%)

      _Schema: InvoiceSchema with fields: merchant_name, date, total_
      _Triggered by commit: abc1234_
      ```

      If no baseline exists (first run), show results without delta section.

      CLI usage:
        python evaluate.py                      # Print results to stdout
        python evaluate.py --output-markdown     # Also write results/pr_comment.md
    </script>

  </scripts>

  <ci_cd>
    <workflow name=".github/workflows/extract-eval.yml">
      Trigger: Pull requests that modify schema.py (the Pydantic extraction schema).

      ```yaml
      name: Extract & Evaluate

      on:
        pull_request:
          paths:
            - 'schema.py'

      jobs:
        extract-and-evaluate:
          runs-on: ubuntu-latest
          permissions:
            pull-requests: write    # Needed to post PR comments

          steps:
            - name: Checkout code
              uses: actions/checkout@v4

            - name: Set up Python
              uses: actions/setup-python@v5
              with:
                python-version: '3.11'

            - name: Install dependencies
              run: pip install -r requirements.txt

            - name: Run extraction
              env:
                LLAMA_CLOUD_API_KEY: ${{ secrets.LLAMA_CLOUD_API_KEY }}
              run: python extract.py

            - name: Run evaluation
              run: python evaluate.py --output-markdown

            - name: Post PR comment
              uses: actions/github-script@v7
              with:
                script: |
                  const fs = require('fs');
                  const comment = fs.readFileSync('results/pr_comment.md', 'utf8');
                  // Find and update existing bot comment, or create new one
                  const { data: comments } = await github.rest.issues.listComments({
                    owner: context.repo.owner,
                    repo: context.repo.repo,
                    issue_number: context.issue.number,
                  });
                  const botComment = comments.find(c =>
                    c.user.type === 'Bot' && c.body.includes('Extraction Evaluation Results')
                  );
                  if (botComment) {
                    await github.rest.issues.updateComment({
                      owner: context.repo.owner,
                      repo: context.repo.repo,
                      comment_id: botComment.id,
                      body: comment,
                    });
                  } else {
                    await github.rest.issues.createComment({
                      owner: context.repo.owner,
                      repo: context.repo.repo,
                      issue_number: context.issue.number,
                      body: comment,
                    });
                  }

            - name: Update baseline on merge
              if: github.event.pull_request.merged == true
              run: |
                cp results/current_metrics.json metrics.json
                git config user.name "github-actions[bot]"
                git config user.email "github-actions[bot]@users.noreply.github.com"
                git add metrics.json
                git commit -m "Update baseline metrics after schema change"
                git push
      ```

      Note on baseline update: The "Update baseline on merge" step only runs when the PR is actually
      merged. This ensures metrics.json always reflects the last merged schema version.

      IMPORTANT: A separate workflow is needed to update metrics.json on merge since the
      pull_request event fires before merge. Use a push trigger on main for this:

      ```yaml
      name: Update Baseline Metrics

      on:
        push:
          branches: [main]
          paths:
            - 'schema.py'

      jobs:
        update-baseline:
          runs-on: ubuntu-latest
          steps:
            - uses: actions/checkout@v4
            - uses: actions/setup-python@v5
              with:
                python-version: '3.11'
            - run: pip install -r requirements.txt
            - name: Run extraction and evaluation
              env:
                LLAMA_CLOUD_API_KEY: ${{ secrets.LLAMA_CLOUD_API_KEY }}
              run: |
                python extract.py
                python evaluate.py
            - name: Commit updated metrics
              run: |
                cp results/current_metrics.json metrics.json
                git config user.name "github-actions[bot]"
                git config user.email "github-actions[bot]@users.noreply.github.com"
                git add metrics.json
                git diff --cached --quiet || git commit -m "Update baseline metrics" && git push
      ```
    </workflow>
  </ci_cd>

  <important_notes>
    <note name="API Key Security">
      The LLAMA_CLOUD_API_KEY must NEVER be committed to the repository.
      - Local development: stored in .env (gitignored)
      - CI/CD: stored as GitHub Actions secret
      - .env.example shows the variable name without the value
    </note>
    <note name="Accuracy Calculation">
      Accuracy is computed at the field level, not the document level.
      - total_fields = number of non-null fields in ground truth across all documents
      - correct_fields = number of fields where extracted value matches ground truth
      - missing fields (null extraction where GT has value) count as INCORRECT
      - overall_accuracy = correct_fields / total_fields
      - Example: 3 files x 3 fields = 9 total fields. If 1 is wrong and 1 is missing: accuracy = 7/9 = 77.8%
    </note>
    <note name="String Comparison">
      Field comparison is case-insensitive with whitespace stripped.
      No fuzzy matching — exact match only after normalization.
      This keeps the evaluation deterministic and simple.
    </note>
    <note name="Webhook Integration">
      Webhook support is built into extract.py, not a separate script. When WEBHOOK_URL env var
      is set, extract.py adds webhook_configurations to every extraction job. This means:
      - Local dev: set WEBHOOK_URL in .env to see webhook payloads on webhook.site
      - CI pipeline: WEBHOOK_URL is not set, so no webhooks fire (just polling)
      - Pre-provisioned URL: https://webhook.site/b87995d5-e449-4074-b311-e0deed2d34dc
    </note>
    <note name="File Format">
      Input files are PDFs in data/. The LlamaExtract SDK handles PDF parsing internally.
      No need for image conversion or OCR libraries in this project.
    </note>
  </important_notes>
